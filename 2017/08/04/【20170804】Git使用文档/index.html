前言、Git的优势（与SVN相比）

1. 强大的分支功能

git最强大的地方就在于它的分支。为了说明方便我可以列举几个场景

  • 今天写了一个功能测试完可以使用，但是同一个文件中有另外一个无足轻重但是有100多行代码的函数还没有写完存在BUG，而且这个函数涉及到的变量和调用还真不少，时间来不及需要发版本了怎么办！创建一个分支用于保存当前代码，再在master分支修改项目发布需要的代码。到时候直接切换到你设置的分支，写到一半的函数就回来了

  • 同一个功能想到了两种方式可以处理，需要实际操作才能知道哪种方法更好。可以在项目上打两个分支分别开发互不影响

这两个小案例当然无法真正展示分支功能的强大，关于分支的说明在下文中还会详细解释

2. 不需要联网提交

相对于SVN的集中式版本控制系统，Git的分布式版本控制系统没有中央服务器，所以提交的时候不需要网络。每个开发者的电脑都是一个完整的版本库，在不提交远程库之前互相独立。如果说两个人同时修改了A文件，那么只需要把这个文件推给对方，就能共享双方的修改内容了。速度飞快

2. 建立仓库后，就可以把项目文件添加到你的路径中

3. 将项目文件添加到仓库中

第一步，用命令git add告诉Git，把文件添加到仓库，这一步可以理解为将文件从我们的工作区添加到Git的暂存区

第二步，用命令git commit告诉Git，把文件提交到仓库。这一步是把暂存区的文件提交到我们本地的版本库中

-m后面输入的是本次提交的注释，便于版本查看

little tips

git add . ：他会监控工作区的状态树，使用它会把工作时的所有变化提交到暂存区，包括文件内容修改(modified)以及新文件(new)，但不包括被删除的文件。

git add -u ：他仅监控已经被add的文件（即tracked file），他会将被修改的文件提交到暂存区。add -u 不会提交新文件（untracked file）。（git add –update的缩写）

git add -A ：是上面两个功能的合集（git add –all的缩写）

4、总结

初始化一个Git仓库，使用git init命令。

添加文件到Git仓库，分两步：

第一步，使用命令git add ，注意，可反复多次使用，添加多个文件；

第二步，使用命令git commit，完成。

三、分支管理

分支可以理解成平行空间，你可以使用git在项目中快速创建无数个平行空间，然后做你想做的事而不影响到其他分支。

master分支是创建仓库时自带的主分支，在团队开发时建议用于发布版本时使用，应该是一个比较稳定的分支，不能随意修改上面的内容。

四、Git基本命令汇总

要随时掌握工作区的状态，使用git status命令。

如果git status告诉你有文件被修改过，用git diff可以查看修改内容。

git reset –hard commit_id。版本退回。commit_id为版本号，可以用git log查看版本号

git reset –hard HEAD 指向的版本就是当前版本

Git鼓励大量使用分支：

查看分支：git branch

创建分支：git branch

切换分支：git checkout

创建+切换分支：git checkout -b

合并某分支到当前分支：git merge

删除分支：git branch -d

4、git 错误 fatal: This operation must be run in a work tree

由于git init –bare 方法创建一个裸仓库，在该仓库无法进行任何git操作，所以抛出错误.
解决方法：在该仓库目录下，新建文件夹，进入该文件夹，执行如下命令 ：

  1. touch Readme
  2. git init
  3. git add Readme
  4. git commit -m ‘initial commit’ Readme

5、 远程有分支，本地没有，push时报错

Updates were rejected because the tip of your current branch is behind
hint: its remote counterpart. Integrate the remote changes (e.g.
hint: ‘git pull …’) before pushing again.
执行git pull

本地与远程代码不同步，需要先把本地代码与远程同步，执行git pull后再进行提交
